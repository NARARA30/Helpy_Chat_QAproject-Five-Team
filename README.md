개인적인 사정으로 프로젝트에 온전히 집중하기 어려운 상황 속에서도, 각자 가능한 시간대에 비동기적으로 꾸준히 참여하며 팀을 유지해온 점이 인상적이었음. 자동화 여부와 관계없이 수동 테스트, 문서 정리 등 각자의 역할을 프로젝트 기여로 인식하고 존중하는 분위기가 잘 형성되어 있었음. 특히 팀장이 팀원들의 상황과 역량을 고려해 역할을 유연하게 조율하며 팀을 이끌고 있어, 전반적인 팀 안정감이 매우 높게 유지되고 있었음.

실무 관점 피드백
결과물보다 과정과 태도를 중시하는 팀이라는 인상이 강했음. 무리해서 모든 것을 해내려 하기보다는, 현재 상황에서 무엇이 가능한지 판단하고 현실적인 선택을 이어간 점이 성숙하게 느껴졌음. 팀 규모가 작고 제약이 많은 환경에서도 불안이나 불만으로 흐르지 않고, 서로를 신뢰하며 끝까지 완주하려는 자세가 팀 전체에 긍정적으로 작용하고 있었음. 질문과 도움 요청을 자연스럽게 받아들이는 문화 역시 큰 강점으로 보임.

한 줄 평
환경과 제약이 많았음에도 불구하고, 책임감과 배려를 바탕으로 끝까지 팀을 지켜낸 팀. 결과보다 태도와 과정이 더욱 인상 깊게 남은 팀이라고 판단함. 초반에 걱정을 굉장히 많이 했던 팀인데, 기대 이상의 성과를 보여줌.

회원가입 테스트에서 이메일 형식 검증, 비밀번호 규칙, 약관 동의 등 다양한 시나리오를 체계적으로 커버한 점이 인상적입니다.

다만, 팀원별 코드가 개인 폴더에 분리되어 통합되지 않은 점이 아쉽습니다. 향후 프로젝트에서는 초기에 공통 구조를 정하고 함께 작업하면 더 좋은 결과를 얻을 수 있을 것입니다.

## 팀원 분들을 위한 깃 기본 가이드

규칙 -> master 브랜치말고 개인 브랜치를 사용하여 명령어 사용해주세요!

## ✅ 1. 변경된 파일 확인

작업한 뒤 반드시 변경 내용을 먼저 확인합니다.

`git status`

빨간색: 스테이지에 올라가지 않은 변경 파일
초록색: 커밋 준비 완료 상태(files staged)

## ✅ 2. 변경 파일 스테이지에 올리기 (add)
특정 파일만 올릴 때

`git add 파일명`

예시:

`git add tests/test_chat_ai.py`

전체 변경 파일 올릴 때 (주의 필요)
`git add .`


>❗ 팀 프로젝트에서는 가급적 git add . 은 조심해서 사용
>불필요한 파일까지 올라갈 수 있으므로 커밋 전에 git status 꼭 확인

## ✅ 3. 커밋 메시지 작성하기 (commit)

반드시 -m 옵션을 사용합니다.

`git commit -m "커밋 메시지 내용"`

예시:

`git commit -m "TC-CB-001, TC-CB-008 테스트 케이스 작성 완료"`

## ✅ 4. 원격 저장소로 올리기 (push)

현재 브랜치가 원격에 연결되어 있다면:

`git push`


처음 생성한 브랜치라 upstream 설정이 필요하면:

`git push -u origin 브랜치명`


예:

`git push -u origin feature/jungeun`
